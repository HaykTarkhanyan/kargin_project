# LEARNINGS

Non-obvious lessons, gotchas, and decisions discovered during work. Append-only — things not derivable from the code itself.

## From the original (pre-rewrite) codebase

- **`pytubefix` scrapes likes via a deeply nested path** (`yt.initial_data['videoPrimaryInfoRenderer']...`). This breaks whenever YouTube changes its page structure. Any rewrite that depends on like counts needs a more durable source (official API or accept-it-might-be-missing).
- **YouTube thumbnails do NOT need pytubefix or any API** — `https://img.youtube.com/vi/<VIDEO_ID>/mqdefault.jpg` works directly. Useful fallback when avoiding YouTube API dependencies.
- **Streamlit multipage apps have no enforced entry point** — users can navigate directly to `pages/X.py`. The original handles this by checking `'df' in st.session_state` and erroring out. Worth designing around in the rewrite.
- **`fuzzywuzzy` is slow on row-by-row scans** — the Telegram bot iterates the entire DataFrame for every text search. Fine at a few hundred rows; would not scale.
- **Bug at `old/pages/1_Search_Episodes.py:349`**: `is_fuzzy=(search_type == "Fuzzy Search")` — but radio values are `"Simple Fuzzy"`/`"Advanced Fuzzy"`, so highlight always falls through to exact-match branch. Fuzzy highlights never render.

## From the metadata-fetch (2026-04-25)

- **Input has 695 unique YouTube video_ids** in `kargin_eng.csv` (`links` column). Zero unparsable URLs in the smoke test.
- **`yt-dlp` is the right tool**, not pytubefix. ~2 sec per video for metadata-only, no audio download.
- **YouTube hid public dislike counts in Dec 2021.** `dislike_count` is `null` for ~all videos. Keep the column for completeness; don't expect data.
- **YouTube's `language` field is unreliable for Armenian content.** Auto-detection often labels Kargin sketches as Russian or Spanish. Don't trust it; use it only as a hint.
- **YouTube's `automatic_captions` dict mostly contains auto-translations, not source captions.** The single key ending in `-orig` (e.g. `ru-orig`) marks the language YouTube actually transcribed in. Everything else is derived. Parse `-orig` to get `auto_caption_source_lang`.
- **For Kargin videos, YouTube auto-captions are often in the wrong language** (Russian/Spanish, due to song intros fooling the language detector). That doesn't make them useless — even mistranslated captions carry real timestamps tied to speech events, which is independently useful as a baseline or for segmentation. Smoke sample (n=3): 2 had auto-captions in `ru`/`es`, 1 had none.
- **Why YouTube misclassifies the language: Kargin sketches frequently open with a Russian or Spanish song as the soundtrack** (per project owner). YouTube's language detector locks onto the song lyrics in the first few seconds and labels the whole video accordingly, even though the dialogue is Armenian. Two consequences for the transcription pipeline:
  1. Gemini STT will likely hit the same failure mode if fed full audio. Either prompt it to focus on speech and transcribe Armenian specifically, or segment out music first.
  2. The `auto_caption_source_lang` column is therefore an indirect signal of "what song is in the intro," not "what language the video is in." Could be useful for a soundtrack-detection feature but is not a reliable language label.
- **No manual captions on any of the smoke-tested videos.** Likely true for the whole set — these are old user uploads.
- **No YouTube chapters either.** Don't expect free segment data from there.
- **Pandas reads empty CSV cells as `NaN`, not `""`.** Caught this in `load_resume()`: `r.get("fetch_error") or ""` evaluated to `NaN`, then `str(NaN)` is `"nan"` (truthy), so the resume logic considered every prior row failed and refetched everything. Always check `pd.isna(value)` explicitly when distinguishing "no error" from "error string". Or use `pd.read_csv(..., keep_default_na=False)` if you want all empties as `""`, but that complicates numeric columns.

## Concurrency & infra (2026-04-25)

- **yt-dlp metadata fetches parallelize cleanly with threads.** I/O-bound, ~2s wall time per call serial → ~0.25s effective with 8 workers. ThreadPoolExecutor + per-call `yt_dlp.YoutubeDL(...)` instances (don't share an instance across threads). 8 workers is fine; haven't tested higher.
- **`uv venv` picked Python 3.11.13 over the system 3.10.10** when `requires-python = ">=3.10"` is set. uv prefers newer Pythons it has access to. Pin a specific minor if reproducibility matters.
- **YouTube rate-limits cookie-less yt-dlp at concurrency, not at total volume.** First metadata run with 8-worker `ThreadPoolExecutor` hit `"Sign in to confirm you're not a bot"` around ~150 cumulative requests. But the 2026-04-26 audio download of all 702 videos serially (single-threaded, ~4 sec/video, ~45 min total, no cookies) finished cleanly with zero failures. Implication: the trigger is concurrent connections, not request count. Fix when needed: `cookiesfrombrowser=("firefox",)` in yt-dlp opts (or `--cookies-from-browser firefox` on the CLI). Chromium-based browsers lock their cookie DB on Windows — Firefox is the easy choice. Plan: prefer single-threaded for bulk yt-dlp operations; only reach for parallelism when single-threaded throughput becomes the actual bottleneck.
- **Chromium-based browsers (Chrome, Edge, Brave) lock their cookie DB while running on Windows** — yt-dlp errors with `"Could not copy Chrome cookie database"` (issue #7271). Workarounds: (a) close all Chrome windows briefly, (b) use Firefox, which doesn't lock, or (c) export cookies via a browser extension and use `--cookies <file>` instead.
- **`taskkill /F /PID ... /T` from Git Bash** mangles the `/F` flag because Git Bash's MSYS path translation interprets it as a Unix path (`F:/`). Wrap in `cmd //c "taskkill /F /PID ..."` to bypass translation.
- **The resume logic gets validated by failure**: a 130-success / 565-fail run was the perfect test for the NaN-handling fix in `load_resume()`. After the fix, restart correctly identified 130 successes and only retried the 565 failures. This is the kind of property you can't unit-test as cheaply as you can validate against real partial-completion state.

## YouTube Data API v3 vs yt-dlp (2026-04-25)

- **API key flow worked first try.** Console signup → enable YouTube Data API v3 → create API key → paste in `.env` as `YOUTUBE_DATA_API_KEY`. Project hits the API as 14 batched calls of 50 video IDs each, ~0.5 sec total wall time vs 25+ minutes for yt-dlp.
- **API is dramatically more reliable than yt-dlp** for bulk metadata: no cookies, no rate-limit anti-bot, batches up to 50 ids per call. Cost: 1 quota unit per `videos.list` call regardless of batch size, so 695 videos = 14 units (free quota is 10,000/day).
- **`yt_declared_language` from the API is much better than yt-dlp's `language` field.** API returns the uploader-declared language (`hy` for Armenian on 578/695 Kargin videos); yt-dlp's auto-detection got fooled by foreign-language song intros and returned `ru`/`es`. The API is the right source for "what language is this video supposed to be in."
- **`captions.list` API costs 50 units per video** (vs 1 for `videos.list`). For 695 videos that's 34,750 units, 3.5x daily quota. Not the right tool for bulk caption fetching here — yt-dlp pulls the same data without quota cost.
- **`contentDetails.caption` from `videos.list` flags MANUAL captions only.** For our archive it returned `False` for all 702 videos — that does NOT mean "no captions exist," it means "no human uploaded captions." Auto-captions can still exist (and do, for many videos), and yt-dlp can fetch them. Don't use this field as a pre-filter for "should we fetch transcripts?"

## YouTube transcript fetch (2026-04-26)

- **`yt-dlp`'s `subtitleslangs` option accepts Python regex patterns.** This is in the docs but easy to miss. Pattern `.*-orig` filters to source-language captions only. Without it, asking for "all" auto-captions writes ~157 files per video — 1 source plus ~156 auto-translation derivatives that are pure noise (Google Translate of the source, not independent transcriptions). With it, you get exactly 1 file per caption-having video.
- **YouTube tags the source-language STT track with a `-orig` suffix** (e.g. `hy-orig`, `tr-orig`). The bare lang code (`hy`, `tr`) is the auto-translated derivative. This convention isn't part of yt-dlp's documented API contract, but it's stable across the current versions and is the only way to distinguish source from translation in the `automatic_captions` dict yt-dlp returns from `extract_info`.
- **YouTube's `contentDetails.caption` field flags MANUAL captions only.** For our archive it's `False` for all 702, but auto-captions exist for 314. Don't use this field as a "should we try fetching transcripts?" pre-filter — it'll incorrectly drop most of the data.
- **Source-lang distribution across 314 caption-having Kargin videos was much wider than any small sample suggested.** Top-source `tr-orig` (92) actually beats `hy-orig` (84). 17 distinct source langs total: tr, hy, en, ru, ro, hi, ar, de, fr, it, es, nl, pl, pt, bg, lv, el. The non-Armenian sources are YouTube STT trying to transcribe foreign-language song intros and giving up after a few keyword fragments. Density is the tell: `hy-orig` files average ~10-12 chars/sec of speech, foreign sources ~0.5-1 chars/sec.
- **Only ~12% of the archive (84 of 702) ends up with usable Armenian text from YouTube.** Another ~33% (231 videos) gives real timestamps tied to speech but garbage text. The remaining ~55% (388 videos) has no captions at all. The "free YouTube salvage" path is real but small; Gemini STT is still the primary transcription path for ~88% of the archive.
- **JSON3 has a lot of presentation cruft** (`pens`, `wsWinStyles`, `wpWinPositions`, plus per-event `wpWinPosId`/`wsWinStyleId`/`wWinId`/`id`). Strip it for any downstream use. Empty cue-only events without `segs` should also be dropped — they contribute nothing to text content. Useful info lives in `events[].segs[].utf8` plus `tStartMs`/`dDurationMs` (in milliseconds; convert to seconds for consistency with `data/youtube_metadata.csv`).

## Gemini STT spike (2026-04-26)

- **"Gemini 3.1 Flash" is no longer a single model.** In 3.1 the Flash family split into Lite (`gemini-3.1-flash-lite-preview`, batch STT, "fast and at scale", free on free tier), Live (`gemini-3.1-flash-live-preview`, real-time bidirectional voice, paid per minute), and TTS (`gemini-3.1-flash-tts-preview`, output-only). For batch transcription of stored audio files, **Lite** is the right model id; Live is 6x more expensive per audio-input token and is overkill for non-streaming work.
- **Lite preview is free on the free tier** for both audio input AND text output. Paid tier rates are $0.50/M tokens audio in, $1.50/M text out. Spike of 17 min audio cost $0.028 paid; full 33.6h archive extrapolates to ~$3.32 paid, free tier should cover it.
- **WebM is NOT in Google's documented supported audio formats** (WAV, MP3, AIFF, AAC, OGG Vorbis, FLAC). Transcode webm/opus → ogg/vorbis with `ffmpeg -i in.webm -vn -c:a libvorbis -q:a 4 out.ogg` before upload. Quality 4 vorbis is plenty for STT and shrinks the upload (~30-50% of the webm size).
- **Audio input token rate is ~26 tokens/sec of audio for Gemini 3.1 Flash Lite.** Useful for cost forecasting: 1 hour of audio ≈ 93k input tokens.
- **`response_mime_type="application/json"`** (passed via `GenerateContentConfig`) is sufficient to get clean JSON without a full pydantic schema — no markdown fences, no preamble. The prompt still has to describe the desired shape; the mime type just constrains the decoder grammar to JSON.
- **Gemini reliably ignores foreign-language song intros when prompted explicitly.** All 3 confounder videos (where YouTube STT was fooled into Romanian/Turkish/Spanish garbage by the song intro) produced correct Armenian transcripts. The working prompt names the failure mode directly: "The video may open with a Russian or Spanish song before the dialogue starts; skip it and only transcribe the spoken Armenian dialogue." Generic "transcribe Armenian speech" without naming the song-intro problem might also work; not tested.
- **Gemini's transcript is qualitatively cleaner than YouTube's hy-orig STT** even on videos where YouTube returned valid Armenian: full punctuation (question marks, periods), proper sentence boundaries, music omitted entirely (vs YouTube interleaving `[երաժշտություն]`), grammatically coherent even where individual words are misheard. YouTube's segmentation is more granular (~2x more events) because it segments on visual subtitle cues; Gemini segments at the clause/sentence level which is better for search.
- **Files API upload is fast and cheap** (~1-2s per file via `client.files.upload(file=path)`). For the 702-file archive at ~1.5 MB average per ogg, total upload time will be ~10-20 min; bottleneck is `generate_content` latency (~5-10s per video) not upload.
- **`PYTHONIOENCODING=utf-8` is required** to print Armenian text in Git Bash on Windows. Default cp1252 charmap raises UnicodeEncodeError on the first Armenian glyph. Set as env var before invoking Python; doesn't affect file I/O which is always controlled via the `encoding="utf-8"` arg.
- **Gemini 3.1 Pro is paid-tier-only** — free-tier requests return 429 immediately with `limit: 0` for all Pro metrics. The 429 fails before upload happens, so probing for "is this model accessible on my tier?" costs nothing. To use Pro, link a billing account via aistudio.google.com → project → Billing.
- **Pro audio-in price (empirically) is ≈ $1.99/M tokens** — Google's pricing page does not list it explicitly. Derived from total Pro spike cost ($0.183) minus text-out cost (10,800 × $12/M = $0.130), divided by audio tokens (26,703). Matches the $2/M estimate from the 2.5 Pro tier.
- **Pro is ~7x slower per inference and ~6.5x more expensive than Lite, but qualitatively more accurate on Armenian.** Specific kinds of fixes Pro made vs Lite on hand-checked sketches: (1) recovers correct opening lines where Lite produced gibberish (`Petros, all day shake your daddies` → `Petrosyan, you're preparing yesterday's contract papers`), (2) uses world knowledge to disambiguate names/places (`Naas darks, Dou Jones` → `Nasdaq, Dow Jones`), (3) gets the right verb register and grammar even when the dialogue is colloquial. The latency cost is "thinking budget" being spent on disambiguation — causally linked to the quality lift, not coincidental.
- **Audio input tokens are model-independent.** Same 26,703 input tokens whether you call Lite or Pro on the same audio. So `tokens_in × per-model rate` is the cost-comparison formula. You can pre-compute audio tokens once and cost-rank every model.
- **Multi-model output namespacing pattern**: `{seq:03d}_{video_id}__{model_short}.json` (double underscore as separator) lets multiple model runs coexist in one directory. Comparison renderer globs `spike_summary__*.json` to auto-discover available models — no list to maintain. Migration of existing single-model outputs is just a rename loop.

## Reconciling data sources (2026-04-25)

- **Use set operations, not counts, when reconciling overlapping data sources.** Counts hide the structure: "200 vs 184" doesn't tell you whether B ⊂ A, B ∩ A = ∅, or anything in between. Set diffs (`A - B`, `B - A`, `A ∩ B`) tell you exactly what's redundant, what's unique, and what to do. Caught a strict subset relationship between two "first 200" sheets this way that count comparisons would have missed.
- **A `done` boolean / curation flag scales better than a "needs_review" sidecar.** When kargin_eng.csv had `done = 1.0` for curated rows, the cleanest way to flag the 6 newly-added unstubbed rows was just `done = 0.0` — pattern continued, no new column needed, downstream filters work unchanged.

## Website MVP — frontend & deploy (2026-06-16)

- **`create-next-app` (Next 16) installs Tailwind v4, which is CSS-first — there is NO `tailwind.config.ts`.** Theming lives in `web/app/globals.css` via `@import "tailwindcss"` + `@theme inline` (declare `--color-*`, `--font-*`, which generate `bg-*`/`text-*`/`font-*` utilities). A plan that `git add`s a `tailwind.config.ts` hard-fails the commit. Verify the installed Tailwind major before assuming v3 syntax.
- **MiniSearch cannot do mid-word substring search.** Its tokenizer splits on whitespace/punctuation and keeps whole tokens, so a pasted mid-quote fragment matches nothing — fatal for quote search. For a small corpus (702 rows) a normalized in-memory `.includes()` scan is simpler, gives true substring, and is sub-millisecond. Don't reach for a search lib reflexively at this scale.
- **Next.js static export → GitHub Pages *project page* needs `basePath = "/<repo>"`.** Gate it on `process.env.GITHUB_ACTIONS === "true"` so local dev/preview stays at root. Intercepting/parallel-route modals do NOT survive `output: 'export'` — ship plain `/sketch/[id]` pages (deep-linkable + shareable) instead of a modal overlay.
- **Rendering all 702 result cards at once is genuinely heavy** — 702 DOM nodes each firing an external `img.youtube.com` request stalls first paint (even a Playwright screenshot timed out at 5s). Fix: cap initial render (~48 + a "load more" button) and use `<img loading="lazy">` (CSS `background-image` does NOT lazy-load). This only surfaced by running the real build in a browser — every unit test was green and the build succeeded.
- **GitHub Pages deployed via Actions serves the artifact as-is — no Jekyll runs**, so `.nojekyll` is belt-and-suspenders, not strictly required. Enable Pages headless with `gh api -X POST repos/OWNER/REPO/pages -f build_type=workflow`; the deploy job 404s ("Ensure GitHub Pages has been enabled") until it's on. The build job runs and validates the whole pipeline even before Pages is enabled.
- **Independent review that *runs the code* beats prose review.** Two adversarial passes (spec, then plan) each caught real bugs only because the reviewer executed/inspected against real data: a `random` sort that returned input order unchanged (untested), a fabricated `title` when a fully-populated `titles` column existed, and the Tailwind-v4 config-file error. "Tests pass + build green" ≠ correct, and ≠ good-at-scale.
- **Actor data is dirty:** 203 distinct raw `main_actors` strings for ~10 real actors (mixed comma/space separators, typos like `Հակյո`→`Հայկո`, `ՄԿո`→`Մկո`, role-words like `ոստիկան` mixed in, and `Ոչ մի` = "nobody" placeholders). Canonicalize by comma-split-first (space-split only when there's no comma, so `փոքր երեխա` isn't torn apart) against a small known-actor allowlist; route non-actors to `rolesNames`.

## Website v2 — EDA, discovery, layout (2026-06-16)

- **One omnipresent entity breaks any "similarity" feature.** Հայko/Мко are in ~76% of sketches, so "related by shared actor" returns nearly the whole corpus → degenerates to "globally most-viewed on every page." Fix = **rarity-weighting**: score a shared actor by `1/frequency`, so a shared *guest* (Հасмик/Аshот) outweighs the ubiquitous duo. Same failure shape as the `Այլ` location catch-all (52% of rows) — exclude or down-weight the dominant value.
- **Armenian name matching: raw substring is unusable; word + case-suffix matching is the fix.** `Անի` as a raw substring matched **365** sketches (94% noise — `պատուհանից`, `անիծել`); declension-aware matching (a *word* equals the name, or starts with it + a known case ending `ին/ի/ից/ով/ում/ը/ն`) gives the real **22**. This solves both over-matching (short names embedded in words) and under-matching (declensions) at once.
- **Next static export + `useSearchParams` REQUIRES a `<Suspense>` boundary or `next build` fails.** Applies to *any* page reading query params — both find-my-name (`?name=`) and the home's `?location=`/`?actor=` seed. The Suspense'd page becomes client-rendered (brief fallback, then hydrates) — fine for these, but it means the home is no longer static-HTML content.
- **Cutting a feature can silently break others.** Dropping "shareable search URLs" left the watch-page tags and Stats labels with nowhere to link (the home couldn't accept a URL filter). Resolved with a *minimal* internal `?location=`/`?actor=` seed — distinct from the full shareable-URL feature. Lesson: when you cut scope, grep for who depended on it.
- **Precompute EDA in Python, render with hand-rolled SVG/CSS.** `stats.json` (aggregates, co-occurrence matrix, bigrams) keeps tokenization server-side and the client lean; SVG/CSS charts (scatter, heatmap, trend, treemap) match the brutalist look and avoid a charting dependency entirely. Bigrams (`ցավդ տանեմ`, `ախпер ջан`) are far more interesting than single-word frequency (which is mostly function words).
- **`npx vitest run` inside a chained *background* shell (no TTY) can report "no tests" spuriously.** A clean foreground re-run found all 14. Don't trust a single "no tests" from a backgrounded gate — re-run in the foreground before concluding the suite is broken.
- **Fuzzy search: index WORDS, not whole strings.** A 1-char typo scored ~0.5 with Fuse.js against each sketch's full searchable string (above a 0.35 threshold → missed), but ~0.17 against **word-level** documents (one Fuse record per word). For short queries against long docs, the query-to-doc length ratio wrecks full-string fuzzy scoring; word-level fixes it.
- **Romanized search = index both forms, don't transliterate the query.** Append `romanize(armenianText)` to each doc's searchable string and match the raw normalized query against the combined text: a Latin query (`tormuz`) hits the romanized part, an Armenian query hits the Armenian part — no query-side transliteration, no language detection. Cheap and symmetric. (Phonetic Eastern-Armenian map; handle the `ու`→`u` digraph before single chars.)
- **Decompose parallel-agent work by FILE, not by feature.** "Romanized search" and "fuzzy search" are different *features* but both rewrite `search.ts`, so separate worktrees would just merge-conflict — they had to share one worktree; the mobile pass (CSS/components) was the only truly independent axis. Each fresh git worktree also needs its own `npm ci` (node_modules is gitignored), so N parallel worktrees = N installs + builds — real local load.
- **Memoize derived search text; don't recompute it per keystroke.** The exact-match path was calling `romanize()` over every sketch's full dialogue on each keystroke (702× per character) — invisible at first, but adding a second transliteration (Cyrillic) would double it. Cache the combined searchable string per sketch in a `WeakMap` keyed by the sketch *object* (safe even if two sketches share an id, unlike an id-keyed Map). Transliteration then runs once, not on every input event. Multi-script search = index all forms (Armenian + Latin + Cyrillic) into one string and match the raw query against it; never transliterate the query or detect its script.
- **Logging from a static site needs an external endpoint, and `sendBeacon` must use `text/plain`.** GitHub Pages can't hold a DB credential, so events POST to a tiny Cloudflare Worker that inserts into Neon (`DATABASE_URL` is a Worker secret). Two gotchas: (1) `navigator.sendBeacon` with an `application/json` Blob triggers a CORS preflight it can't perform → send the body as a **`text/plain` Blob** (CORS-safelisted) and `JSON.parse` it on the worker; (2) insert a jsonb column with an explicit `::jsonb` cast on the param, or Postgres errors "column is jsonb but expression is text". Ship logging **dormant** (no-op until an env var sets the endpoint) so it's decoupled from provisioning the backend.
- **Dark theme on a token-based design is cheap; the leaks are the hardcoded colors.** Flipping the CSS variables in a `.dark` class re-themes everything that uses the tokens (`bg-paper`, `text-ink`, `.k-shadow` → light offset shadow on dark, motif preserved) — for free. The work is the *non-token* colors: `bg-white` surfaces (override once: `.dark .bg-white{...}`, which wins as unlayered CSS over Tailwind's layered utility), SVG presentation attrs hardcoded to ink (`stroke="#1A1410"` → `className="stroke-ink"` so they flip), and inline pastel cells (pin an explicit dark text color so they're readable in both themes). A pre-paint inline `<script>` reading localStorage avoids the flash; `suppressHydrationWarning` on `<html>` silences the class mismatch.
- **Audit parallel-agent deviations — a broken environment can make an agent "fix" the wrong thing.** Two worktree agents each ran `npm ci`, filling the disk to 100%; one got a corrupt `next` install, concluded "the type check is broken," and slipped `typescript:{ignoreBuildErrors:true}` into next.config (disabling type-checking site-wide). The real fix was reinstalling. Lesson: when an agent reports a "pre-existing framework bug" that contradicts known-green history, suspect its environment, and re-verify any global escape-hatch it added against a clean build before merging. Also: N parallel worktrees = N full `node_modules` — real disk pressure on a constrained machine; prune worktrees promptly.
